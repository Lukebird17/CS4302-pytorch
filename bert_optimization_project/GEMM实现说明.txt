三、核心优化技术详解
================================================================================

【优化1: Shared Memory Tiling】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

问题: 直接从Global Memory读取非常慢（400-800 cycles）

解决: 将数据分块加载到Shared Memory（~20 cycles）

示意图:
  Global Memory (慢)
       ↓ 加载一次
  Shared Memory (快)
       ↓ 重复使用32次
  Registers (超快)
       ↓ 计算
  结果

代码:
```cuda
__shared__ float As[TILE_SIZE][TILE_SIZE];
__shared__ float Bs[TILE_SIZE][TILE_SIZE];

// 加载到shared memory
As[ty][tx] = A[row * K + t * TILE_SIZE + tx];
Bs[ty][tx] = B[(t * TILE_SIZE + ty) * N + col];

__syncthreads();  // 等待所有线程加载完成

// 从shared memory计算（快！）
for (int k = 0; k < TILE_SIZE; k++) {
    sum += As[ty][k] * Bs[k][tx];
}
```

效果: 减少全局内存访问 ~32倍


【优化2: Register Tiling】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

问题: 每个线程只计算1个输出，寄存器利用率低

解决: 每个线程计算8x8=64个输出，存在寄存器中

对比:
  基础版本: 1个线程 → 1个输出 → 需要32*32=1024个线程
  优化版本: 1个线程 → 64个输出 → 只需16个线程

代码:
```cuda
// 寄存器数组存储64个输出
float accum[8][8] = {0.0f};

// 计算循环
for (int i = 0; i < 8; i++) {
    for (int j = 0; j < 8; j++) {
        accum[i][j] += As[...][k] * Bs[k][...];
    }
}
```

效果: 减少线程数 → 提高occupancy → 隐藏延迟


【优化3: Memory Coalescing（合并访问）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

问题: 不连续的内存访问需要多次事务

解决: 确保warp内32个线程访问连续的128字节

优化前（跨步访问）:
  Thread 0: A[0]
  Thread 1: A[K]      ← 不连续！
  Thread 2: A[2*K]
  ...

优化后（连续访问）:
  Thread 0: A[0]
  Thread 1: A[1]      ← 连续！可以合并
  Thread 2: A[2]
  ...

代码技巧:
```cuda
// 确保连续访问
int tid = threadIdx.y * blockDim.x + threadIdx.x;
int row = tid / TILE_SIZE;
int col = tid % TILE_SIZE;

// 加载时按行展开
for (int i = tid; i < TILE_SIZE * TILE_SIZE; i += num_threads) {
    int r = i / TILE_SIZE;
    int c = i % TILE_SIZE;
    As[r][c] = A[...];  // 连续访问
}
```

效果: 4-32个独立事务 → 1个合并事务


【优化4: Bank Conflict避免】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

问题: Shared Memory有32个Bank，同时访问同一Bank会冲突

解决: Padding或重新排列访问模式

优化前:
```cuda
__shared__ float As[32][32];  // 每行32个元素
// 访问As[0][0], As[1][0], ... As[31][0]
// → 全部访问Bank 0！32-way冲突！
```

优化后:
```cuda
__shared__ float As[32][33];  // 每行33个元素（Padding）
// 访问As[0][0], As[1][0], ... As[31][0]
// → 访问Bank 0, 1, 2, ... 31  无冲突！
```

效果: 访问延迟降低 ~32倍


【优化5: 循环展开（针对BERT的K=768）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

问题: 动态循环有分支开销和指令依赖

解决: 编译时展开所有循环

代码:
```cuda
// 优化前: 动态循环
for (int k = 0; k < K; k++) {
    sum += As[ty][k] * Bs[k][tx];
}

// 优化后: 完全展开（K=768已知）
#pragma unroll 24  // 768/32 = 24
for (int k = 0; k < 768; k += 32) {
    #pragma unroll 32
    for (int kk = 0; kk < 32; kk++) {
        sum += As[ty][k+kk] * Bs[k+kk][tx];
    }
}
```

效果: 无分支开销 + 指令级并行


【优化6: float4向量化】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

问题: 每次只加载1个float（4 bytes）

解决: 一次加载4个float（16 bytes）

代码:
```cuda
// 优化前
float val = A[idx];

// 优化后
float4 vals = *reinterpret_cast<const float4*>(&A[idx]);
// vals.x, vals.y, vals.z, vals.w 一次性读取
```

效果: 内存带宽利用率提高 4倍


================================================================================
四、融合优化: GEMM + Bias + GELU
================================================================================

【为什么要融合？】
  分离实现:
    GEMM: C = A * B
    Add:  C = C + bias
    GELU: C = GELU(C)
    → 3次内存往返！

  融合实现:
    一个kernel: C = GELU(A * B + bias)
    → 1次内存往返！

【实现】
```cuda
__global__ void gemm_bias_gelu_kernel_768(...) {
    // 1. GEMM
    float sum = 0.0f;
    for (int k = 0; k < 768; k++) {
        sum += As[ty][k] * Bs[k][tx];
    }
    
    // 2. 立即加bias和GELU（无需写回内存）
    sum += bias[col];
    sum = fast_gelu(sum);  // 寄存器操作，超快！
    
    // 3. 一次性写回
    C[row * N + col] = sum;
}
```

【效果】
- 减少2次内存往返
- 预期提升: 5-10%（在GEMM已优化的基础上）

七、常见问题
================================================================================

Q1: 为什么我的GEMM比cuBLAS慢很多？
A:  正常！cuBLAS极度优化过，我们的目标是达到50-80%即可

Q2: 如何进一步优化？
A:  - 使用Tensor Core（FP16/INT8）
    - Warp-level primitives (wmma)
    - 更激进的Tile参数调优
    - CUTLASS库（NVIDIA开源的GEMM模板）

Q3: Bank Conflict怎么检查？
A:  nvprof --metrics shared_load_transactions_per_request

Q4: 为什么大batch效果更好？
A:  - 更高的并行度
    - 更好的内存带宽利用
    - Kernel启动开销占比更小
