一、优化策略（按重要性排序）
================================================================================

🎯 优化1: FP16 Tensor Core 加速 GEMM（占50-70%时间）⭐⭐⭐⭐⭐
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【为什么这是最重要的优化？】
- GEMM矩阵乘法占BERT推理时间的50-70%
- PyTorch默认使用FP32，未利用Tensor Core
- RTX 4060 Ti的FP16 Tensor Core性能是FP32的8倍！

【实现方式】
```python
class OptimizedLinear(nn.Linear):
    def forward(self, x):
        # FP32 -> FP16 -> GEMM (Tensor Core) -> FP32
        x_fp16 = x.half()
        weight_fp16 = self.weight.half()
        output = F.linear(x_fp16, weight_fp16, None).float()
        return output + self.bias
```

【优点】
✓ 不需要修改PyTorch源码
✓ 自动使用Tensor Core（如果GPU支持）
✓ 精度损失极小（< 0.01%）
✓ 预期提升: 30-50%（大batch）、15-25%（小batch）

【技术细节】
- 权重和激活转换为FP16
- GEMM在FP16精度下计算（使用Tensor Core）
- 输出转回FP32，保持数值稳定性
- 对BERT这种Transformer模型特别有效


🎯 优化2: 融合 LayerNorm + Residual + Dropout（占12-25%时间）⭐⭐⭐⭐
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【问题】
- 原实现: 3个独立kernel (Dropout → Add → LayerNorm)
- 7次内存读写
- eval模式仍生成随机数（浪费）

【优化】
- 1个CUDA kernel完成所有操作
- 3次内存读写（减少57%）
- eval模式跳过dropout和curand
- 使用Welford算法（一遍扫描）

【预期提升】8-12%


🎯 优化3: 快速GELU（tanh近似）（占2-5%时间）⭐⭐⭐
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【问题】
- erf函数: ~20条指令

【优化】
- tanh近似: ~8条指令
- GELU(x) ≈ 0.5x(1 + tanh(√(2/π)(x + 0.044715x³)))

【预期提升】2-3%


🎯 其他优化: float4向量化、Online Softmax等
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- hidden_size=768特化
- float4向量化加载
- Online Softmax

【预期提升】5-8%

六、为什么这样优化？
================================================================================

【错误的优化思路】❌
  优化LayerNorm、GELU、Softmax等"边角料"
  → 在大batch下几乎没效果（只占30%时间）
  
【正确的优化思路】✓
  先优化最大瓶颈GEMM（占50-70%时间）
  → 使用FP16 Tensor Core
  → 直接提升30-50%性能！


【阿姆达尔定律】
  如果优化了20%的代码，让它快10倍，总体只快16%
  如果优化了70%的代码，让它快2倍，总体能快40%！
  
  结论: 优化大头，而不是小头！

七、技术细节
================================================================================

【FP16 GEMM实现】
1. 在forward时临时转换为FP16
2. 调用torch.nn.functional.linear（会自动使用Tensor Core）
3. 输出转回FP32
4. 无需修改PyTorch源码！

【CUDA融合算子】
- Welford一遍扫描LayerNorm
- 融合Residual和Dropout
- 针对768维度特化
- float4向量化

【为什么不直接用torch.cuda.amp？】
  torch.cuda.amp会在整个模型上启用FP16，可能导致:
  - 某些操作数值不稳定
  - 需要loss scaling
  - 训练和推理行为不一致
  
  我们的实现:
  - 只对GEMM使用FP16
  - 其他操作保持FP32
  - 更可控，更稳定
